# live 반영 롤백 절차

## 목적
live 반영 직후 문제가 생겼을 때
- 어떤 파일을 먼저 확인해야 하는지
- 어느 배포본으로 되돌릴지
- DB/로컬 캐시/업데이트 공지 흐름을 어떻게 처리할지
빠르게 판단하기 위한 운영용 메모입니다.

## 1. 먼저 확인할 것
1. `D:\거래플랜\테스트 시행\기록\deploy-...\live-preflight.md`
2. `D:\거래플랜\테스트 시행\기록\deploy-...\live-postflight.md`
3. `D:\거래플랜\배포\업데이트\manifest\stable.json`
4. `D:\거래플랜\배포\관리자용\버전보관`

## 2. 롤백 기준
아래 중 하나면 롤백 검토
- live 반영 후 앱이 시작조차 안 됨
- 필수 업데이트 루프 / 시작 차단이 광범위하게 발생
- 동일 데이터 저장/동기화가 여러 PC에서 반복 실패
- 렌탈/거래처/품목 핵심 화면이 열리지 않음
- 서버/데스크톱 둘 중 하나만 반영되어 버전 불일치가 심함

## 3. 데스크톱 롤백 기본 원칙
- 가장 최근 정상 버전의 설치 패키지와 업데이트 zip 은 `배포\관리자용\버전보관` 기준으로 선택
- manifest `stable.json` 은 롤백 대상 버전에 맞춰 다시 배포
- 필수 업데이트를 걸었던 경우 `minimumSupportedVersion` 도 함께 롤백 대상 버전에 맞춰 재작성
- 사용자가 이미 새 버전 설치 후 문제가 난 경우, 롤백 설치 패키지로 재설치 안내

## 4. 데이터 롤백 기본 원칙
- 서버 데이터 문제면 서버 백업 기준으로 복구 여부를 먼저 판단
- 클라이언트 로컬 캐시 문제면 앱 시작 후 전체 재동기화 / 무결성 점검으로 복구를 우선 시도
- 로컬 캐시만으로 해결 안 되면 `AppData\backup` 또는 서버 백업 사용
- DB 롤백은 데스크톱 바이너리 롤백과 분리해서 판단하지 말고 반드시 세트로 검토

## 5. 권장 순서
1. 반영 직전/직후 로그 확인
2. 문제 범위 판단(특정 PC / 전체 PC / 특정 계정 / 전체 계정)
3. 캐시 문제면 전체 재동기화 먼저
4. 바이너리 문제면 이전 stable 버전 manifest + 설치 패키지 재배포
5. 데이터 손상 문제면 백업 기준 복구 검토
6. 롤백 후 다시 `Invoke-LiveReleaseReadinessCheck.ps1` 와 기본 smoke 테스트 수행

## 6. 메모
- manifest 만 되돌리고 설치 패키지를 안 맞추면 또 다른 혼선이 생길 수 있음
- 필수 업데이트였던 버전은 `minimumSupportedVersion` 을 함께 되돌리지 않으면 구버전 접속이 계속 차단될 수 있음
- 롤백 후에도 사용자가 이미 받은 로컬 dirty 데이터는 동기화 정책 영향을 받을 수 있으므로 재로그인/전체 재동기화 안내가 필요할 수 있음
