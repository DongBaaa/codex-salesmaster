## 8. 남은 위험

1. Android stable `0.2.24`는 아직 기존 배포본을 가리키므로, 모바일 사용자는 배포 전까지 로그인 실패 위험이 남아 있다.
2. 공식 Android 업데이트 APK 생성은 `D:\거래플랜\Mobile\GeoraePlan.Mobile.App\android-signing.local.json` 부재와 실제 keystore 부재로 현재 차단된다. 프로젝트 범위에서는 `android-signing.example.json`만 발견되었고, 기존 사용자 업데이트를 보장하려면 기존 배포본과 같은 서명키 설정이 필요하다.
3. Android 운영 Release 기본 AOT 직접 publish는 현재 로컬 경로에서 `response file can not be read` 오류가 발생했다. 공식 배포 단계에서는 기존 배포 스크립트 옵션으로 재검증해야 한다.
4. 운영 계정별 scope 누수는 자동 로컬 테스트와 운영 게이트 계정 범위 회귀로 점검했지만, 추가 실사용 계정/회사/담당지점 샘플의 read-only SQL 대조는 남아 있다.
5. WPF DatePicker/알림창은 source guard로 범위를 넓혔지만, 실제 화면 캡처 기반 전수 UIA 검사는 아직 남아 있다.
6. 장시간 자동 새로고침과 사용 중 선택/입력값 보존은 sync 회귀 테스트로 일부 보장되지만, 실제 장시간 관찰은 남아 있다.

## 9. 후속 권장 작업

1. 기존 Android 업데이트 서명키 설정을 준비한 뒤 Android 버전을 `0.2.25` 이상으로 올리고 APK/업데이트 자산을 재생성한다.
2. Android 공식 패키지 생성 전에는 `Build-GeoraePlanAndroidApk.ps1 -SkipDeploymentCopy`로 배포 폴더 변경 없이 서명/패키징을 먼저 검증한다.
3. 생성된 Android 패키지를 emulator 또는 실기기에서 다시 smoke PASS 확인 후 Linux PC 반영과 manifest 확인을 진행한다.
4. 추가 실사용 계정/회사/담당지점 샘플로 read-only SQL/API 대조를 수행한다.
5. `billingAdvanceMode` 업무 정의 확정 후 필요 시 선불/후불 기간 계산 테스트를 추가한다.
6. WPF UIA 또는 screenshot 기반 DatePicker/알림창 전수 검사 자동화를 추가한다.
