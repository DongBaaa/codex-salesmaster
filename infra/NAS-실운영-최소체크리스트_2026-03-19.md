## 0. 2026-03-19 현재 확인된 상태

이 PC에서 확인한 현재 상태는 아래와 같다.

- `api.example.invalid` DNS 해석 실패
- `192.0.2.10:18082` 연결 실패
- `https://api.example.invalid/healthz` 를 NAS IP로 강제 연결하면 Synology nginx 404 반환
- `http://192.0.2.10/`, `https://192.0.2.10/` 는 Synology 기본/보호 페이지 응답
- `\\192.0.2.10\docker\georaeplan` SMB 경로 접근 실패

즉, NAS 자체는 살아 있지만 **거래플랜 API가 실제로 외부에서 보이도록 배포된 상태는 아직 아님**.

## 3. NAS `.env` 작성

NAS에서 실제 운영값으로 `ops\.env` 를 만든다.

권장 시작 파일:

```text
\\192.0.2.10\docker\georaeplan\ops\.env.api.example.invalid.example
```

최소 필수 수정값:

- `POSTGRES_PASSWORD`
- `JWT_SIGNING_KEY`
- `PUBLIC_BASE_URL`
- `SECURITY_REQUIRE_HTTPS_FORWARDED_PROTO`
- `SEED_USERS_ADMIN_PASSWORD`

핵심 기본값:

```env
NAS_ROOT_PATH=/volume1/docker/georaeplan
HOST_BIND_IP=127.0.0.1
POSTGRES_DB=georaeplan
POSTGRES_USER=georaeplan
POSTGRES_HOST_PORT=15434
API_HOST_PORT=18082
JWT_AUDIENCE=georaeplan-client
PUBLIC_BASE_URL=https://api.example.invalid
TZ=Asia/Seoul
```

### 확인 항목
- [ ] `HOST_BIND_IP=127.0.0.1` 로 되어 있는가
- [ ] `Database__EnableSqliteFallback=false` 환경으로 동작하게 되는가
- [ ] `POSTGRES_PASSWORD` 가 강한 값으로 변경되었는가
- [ ] `JWT_SIGNING_KEY` 가 충분히 긴 랜덤 문자열로 변경되었는가
- [ ] `SECURITY_REQUIRE_HTTPS_FORWARDED_PROTO=true` 인가
- [ ] `SEED_USERS_ADMIN_PASSWORD` 가 빈값/예시값이 아닌 실제 강한 값인가
- [ ] `PUBLIC_BASE_URL` 이 실제 공개 주소와 일치하는가

## 4. 서버 publish 및 NAS `app/live` 반영

로컬 PC에서 아래 명령 실행:


## 7. 리버스 프록시 / 도메인 연결

현재 확인상 `api.example.invalid` 은 이 PC에서 DNS 해석이 안 되고, NAS 강제 연결 시 Synology nginx 404 이다. 이는 **도메인 또는 reverse proxy 라우팅이 아직 완료되지 않았다는 뜻**이다.

따라서 아래를 맞춰야 한다.

### DNS
- [ ] `api.example.invalid` 이 NAS 공인 IP 또는 내부 테스트용 IP로 해석되는가

### Reverse Proxy
- [ ] 호스트: `api.example.invalid`
- [ ] 스킴: HTTPS
- [ ] 대상: `http://127.0.0.1:18082`
- [ ] `/healthz` 포함 전체 경로가 API로 전달되는가

### 외부 점검
외부 PC 또는 현재 PC에서:

```powershell
curl https://api.example.invalid/healthz
```

### 성공 기준
- [ ] Synology 404/403 이 아닌 API JSON 응답이 오는가
- [ ] 인증서 오류가 없는가

## 8. 로그인 확인

서버가 뜨면 기본 seed 계정으로 먼저 확인 가능하다.

- 계정: `admin`
- 비밀번호: `<운영 비밀번호>`

> 운영 전에는 반드시 비밀번호를 변경하는 것을 권장.

간단 API 테스트:

```powershell
$body = @{ username = 'admin'; password = '<운영 비밀번호>' } | ConvertTo-Json
Invoke-RestMethod -Uri 'https://api.example.invalid/auth/login' -Method Post -ContentType 'application/json' -Body $body
```

### 확인 항목
- [ ] `/auth/login` 응답이 200 인가
- [ ] JWT 발급이 되는가
- [ ] `Audience=georaeplan-client` 기준으로 정상 토큰인가

## 11. 최종 결론 체크

아래 4개가 모두 만족되면, **PC 버전과 Android APK는 같은 NAS 데이터를 공유하는 상태**라고 판단할 수 있다.

- [ ] PC와 Android가 모두 같은 `Api.BaseUrl` / 서버 주소를 사용한다
- [ ] 둘 다 같은 NAS API (`api.example.invalid`) 로 로그인한다
- [ ] NAS API가 같은 PostgreSQL DB 를 사용한다
- [ ] 한쪽에서 만든/삭제/복원한 데이터가 다른 쪽에서도 보인다

## 12. 현재 우선순위 1위

지금 가장 먼저 해야 할 것은 아래 3개다.

1. `api.example.invalid` DNS 또는 내부 테스트용 호스트 연결 정상화
2. NAS reverse proxy 를 `127.0.0.1:18082` 로 연결
3. NAS `georaeplan` 컨테이너 실제 기동 및 `/healthz` 성공 확인

이 3개가 끝나야, 그 다음 **실제 NAS 기준 PC/Android 공동 사용 검증**이 가능하다.
