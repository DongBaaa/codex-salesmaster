# 원격 main 병합 전략 (2026-03-20)

## 현재 상태
- 현재 작업 브랜치: `codex/current-state-20260320`
- 현재 커밋: `d13ed03`
- 원격 기본 브랜치: `origin/main`
- 확인 결과: `origin/main` 과 현재 브랜치는 **merge base가 없는 서로 다른 역사(unrelated histories)** 입니다.

## 왜 바로 main 에 병합하면 위험한가
1. `git merge-base HEAD origin/main` 결과가 없어서 일반적인 fast-forward / rebase 흐름이 불가능합니다.
2. 두 이력 모두 같은 폴더(`Desktop`, `Server`, `Shared`, `README.md`, `배포`)를 포함하고 있어 단순 `--allow-unrelated-histories` 병합 시 대량 충돌 가능성이 큽니다.
3. 현재 작업 브랜치에는 설치 패키지, 테스트 산출물, 아이콘 자산, NAS 배포 스크립트 등 운영 산출물이 같이 들어 있어 그대로 합치면 원격 main 정리가 더 어려워집니다.

## 권장 전략
### 권장안 A: origin/main 기반 통합 브랜치를 새로 만들어 소스만 선별 병합
1. `origin/main` 기준 새 통합 브랜치 생성
   - 예: `codex/integrate-origin-main`
2. 현재 작업 브랜치에서 **소스/문서/스크립트만** 선별 반영
   - 포함 권장: `Desktop`, `Mobile`, `Server`, `Shared`, `Updater`, `infra`, `tools`, `README.md`, `거래플랜.sln`, `업데이트 내역.md`
   - 제외 권장: `배포/설치패키지/*.zip`, `*.exe`, `*.msi`, `배포/테스트`의 생성 산출물, `.tmp`, bugreport zip
3. 플랫폼별 검증
   - `dotnet build D:\거래플랜\거래플랜.sln -c Release`
   - Android APK/AAB 빌드
   - NAS 배포 스크립트 dry-run 또는 실배포 검증
4. 검증 후 PR을 `origin/main` 으로 생성

### 권장안 B: 기능 단위 cherry-pick 재적용
현재 브랜치의 큰 누적 커밋을 그대로 합치지 않고, 아래 순서로 기능 단위를 다시 나누는 방식입니다.
1. 리네이밍/브랜딩
2. NAS 배포/업데이트 구조
3. 모바일 기능
4. 보안/권한/office scope
5. sync 복구/AAB 빌드

장점:
- 리뷰가 쉬움
- 문제 발생 시 되돌리기 쉬움

단점:
- 시간이 더 걸림

## 비권장 전략
### origin/main 에 바로 `merge --allow-unrelated-histories`
- 가능은 하지만 충돌 범위가 너무 넓고, 생성 산출물까지 같이 섞일 위험이 큽니다.
- 운영 파일과 소스 파일 경계가 흐려져 추후 유지보수가 어려워집니다.

## 이번 정리에서 같이 반영한 사항
- `.gitignore` 를 보강해 임시 파일과 테스트 산출물이 다시 워킹트리를 오염시키지 않게 정리
- `배포/테스트/PC`, `배포/테스트/모바일` 의 경량 실행 스크립트만 추적 가능하도록 조정
- `Run-PC.cmd` 를 실행 파일 탐색형으로 보강해 테스트 폴더의 exe 이름 차이에도 대응하도록 수정
